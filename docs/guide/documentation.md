# JustMemo 文档体系与维护规范

> 最后更新：2026-06-09
> 状态：当前项目文档结构基线

## 1. 作用

这份文档定义 JustMemo 文档库的组织方式，解决四个问题：

1. 文档应该按什么层级划分。
2. 同一种信息应该放在哪一类目录。
3. 不同文件类型应该用什么命名和格式。
4. 文档变更时，应该更新哪一份，而不是不断新增重复文件。

## 2. 当前结论

当前 `/docs` 的四个一级目录整体上是合理的，不需要立刻做大规模重组：

- `core`：系统级事实
- `interface`：视觉与交互规范
- `features`：业务功能说明
- `guide`：流程、规范、测试、技术栈与文档治理

也就是说，现阶段最需要的不是“重命名整个文档树”，而是：

1. 明确每个目录的边界。
2. 统一文档格式。
3. 控制单文件体积，避免继续把过多主题堆进同一篇文档。

## 3. 文档层级

建议按四层理解整个项目文档体系。

### L0：仓库级文档

放在仓库根目录或独立子目录下，用于解释整个项目或某个子系统目录：

- `README.md`
- `CLAUDE.md`（符号链接 → `AGENTS.md`）
- `scripts/README.md`
- `supabase/README.md`

适合承载：

- 项目概览
- AI agent 工作指令
- 特定目录的使用说明

### L0b：不进入 Git 的过程文档

开发过程中产生的临时文件，属于过程产物，不应进入版本库。这些文件放在 `docs/` 下但已加入 `.gitignore`：

- `docs/reviews/` — 代码审查报告
- `docs/planning/` — 开发计划、配置参考（如 Vercel 环境变量清单）
- 未来新增的过程文档统一放在 `docs/proc/` 或 `docs/dev/`，并通过 `.gitignore` 排除

判断一条信息是否应该进入 Git，问自己：**三个月后、一个新开发者接手时，这条信息还有参考价值吗？** 如果答案是"否"，就不要 commit。

### L1：领域级文档目录

放在 `/docs` 下，按“信息类型”而不是“作者习惯”分类。

#### `docs/core/`

存放系统级约束与底层事实：

- 架构
- API
- 数据库
- 安全

特点：

- 这类文档描述的是“系统如何工作”
- 它们通常跨多个功能模块
- 一旦变化，影响面较大

#### `docs/interface/`

存放视觉系统与交互规则：

- 设计系统
- 动效规范
- 交互行为

特点：

- 这类文档描述“界面如何表现”
- 更偏前端体验与设计约束

#### `docs/features/`

存放业务功能说明：

- 某个页面
- 某类功能
- 某个运营能力

特点：

- 这类文档描述“用户能做什么”
- 更偏产品实现与模块说明

#### `docs/guide/`

存放开发流程和治理规则：

- 工程规范
- 测试策略
- 技术栈
- 文档规范本身

特点：

- 这类文档描述“团队怎么开发、怎么维护”

### L2：文件级类型

同样是 Markdown 文件，也应区分职责。

推荐使用以下类型：

#### Index

入口索引型文档，只负责导航。

例如：

- `docs/README.md`

规则：

- 不堆积长篇实现细节
- 只做目录说明、阅读顺序、跳转入口

#### Spec

规格说明型文档，描述稳定规则与约束。

例如：

- `docs/core/api.md`
- `docs/core/database.md`
- `docs/core/security.md`
- `docs/interface/design.md`

规则：

- 描述“当前真实规则”
- 避免写成长篇开发日志
- 优先保持可查证、可执行

#### Guide

指南型文档，描述流程、实践和使用方式。

例如：

- `docs/guide/testing.md`
- `docs/guide/standards.md`
- `docs/features/features-guide.md`

规则：

- 可以有步骤、建议、清单
- 允许带一定解释性内容

#### Module Note

模块专项文档，描述某个独立功能。

例如：

- `docs/features/usage-monitoring.md`

规则：

- 聚焦一个主题
- 不要扩展成“整个系统大全”

### L3：章节级组织

单篇文档内部也需要统一结构。

推荐顺序：

1. 标题
2. 元信息
3. 这篇文档的作用
4. 核心规则或结构
5. 关键边界 / 例外
6. 关联文档

## 4. 当前目录与文件映射

### `docs/core/`

- `architecture.md`：业务逻辑架构与状态流
- `api.md`：Server Actions、查询接口、动态 API 调用约束
- `database.md`：数据结构、RLS、RPC、索引
- `security.md`：私密 Memo 规则与权限边界

结论：

- 分类合理
- 但 `architecture.md` 更新频率明显落后于其他核心文档，后续需要更严格同步

### `docs/interface/`

- `design.md`：设计系统与交互规范

结论：

- 分类合理
- 当前只有单一设计系统文档，后续拆分为 system + interactions 时再更新此处

### `docs/features/`

- `features-guide.md`：功能总览与入口索引
- `gallery.md`：画廊能力
- `map.md`：地图能力
- `timeline.md`：时间轴、归档与那年今日
- `selection.md`：多选与批量操作
- `content-experience.md`：正文解析、链接预览、邮件交互与缓存预热
- `backup-recovery.md`：备份与恢复流程
- `export-spec.md`：海报生成与二维码分享规格
- `usage-monitoring.md`：管理员侧监控功能专项说明

结论：

- 目录方向正确
- 当前已经完成从“大而全”总文档向“总览 + 专题页”的拆分
- 后续继续新增功能时，应优先补充已有专题页，避免重新把总览页写胖

### `docs/guide/`

- `standards.md`：工程规范
- `tech-stack.md`：技术栈
- `testing.md`：测试策略
- `documentation.md`：文档体系与维护规范

结论：

- 分类合理
- 这是当前最稳定的一组

## 5. 文件命名规范

所有文档文件名统一使用：

- 小写
- kebab-case
- 不带日期
- 不带版本号

正确示例：

- `documentation.md`
- `usage-monitoring.md`
- `tech-stack.md`

不建议：

- `DocsGuideV2.md`
- `2026-04-security-update.md`
- `final_doc.md`

原因：

- 日期和版本应写在文档内容里，而不是文件名里
- 文件路径应该稳定，避免频繁改名导致引用失效

## 6. 文档头部格式

建议所有 `/docs` 下文档都统一为：

```md
# 文档标题

> 最后更新：YYYY-MM-DD
> 状态：可选，写“已实现 / 草案 / 待确认 / 仅规划”
```

说明：

- `最后更新` 必须写绝对日期
- `状态` 不是强制，但对规则类文档非常有帮助

## 7. 文档内容格式

### 标题层级

- 优先控制在 `H1 -> H2 -> H3`
- 尽量不要使用过深层级

### 列表风格

- 同一篇文档里尽量统一使用同一种列表风格
- 说明类内容优先短句
- 长说明优先拆段，不要一整段堆满多个主题

### 代码块

- SQL、TS、JSON 等代码块尽量带语言标识

### 表格

- 适合用来放字段、参数、目录映射、职责边界
- 如果表格内容太长，应改成分段列表

## 8. 新文档应该放哪

新增文档时，优先按“内容属性”归类。

### 放进 `core`

如果它回答的是：

- 系统怎么工作
- 权限怎么判断
- 数据怎么存
- 接口怎么约定

### 放进 `interface`

如果它回答的是：

- 页面怎么展示
- 动效怎么执行
- 设计 tokens 和组件视觉怎么统一

### 放进 `features`

如果它回答的是：

- 某个业务能力怎么设计
- 某个页面对用户提供什么功能
- 某个模块有哪些交互边界

### 放进 `guide`

如果它回答的是：

- 团队应该怎么开发
- 怎么测试
- 怎么维护
- 文档体系本身怎么治理

## 9. 什么时候新建文档，什么时候扩展旧文档

优先扩展旧文档，而不是默认新建。

应该扩展旧文档的情况：

- 只是给现有规则补充一个新边界
- 只是补充已有模块的新字段、新流程
- 内容仍然属于同一个主题

应该新建文档的情况：

- 新主题已经和旧文档不是一个层级
- 旧文档篇幅过大，阅读成本明显上升
- 新内容需要独立维护者或独立变更节奏

## 10. 当前最需要的调整

这里给出的是“结构建议”，不是必须立刻执行的大重构。

### 建议 1：让 `docs/README.md` 永远只做索引

这是本次已经完成的调整。

原因：

- 入口文档不应该同时承担导航、规则、架构解释三种职责
- 否则它会很快变成第二份 `features-guide.md`

### 建议 2：继续保持 `features-guide.md` 的索引属性

这一项已经完成：

- `features-guide.md` 已退化为总览页
- 画廊、地图、时间轴、多选、内容体验已经拆成独立专题文档

后续新增功能时，优先判断：

- 是否可以归入现有专题页
- 是否已经足够大到需要独立文档

### 建议 3：给运维 / 管理类功能预留子分类

如果管理员功能继续增加，建议未来引入：

- `docs/features/admin/`

届时 `usage-monitoring.md` 可以迁入该目录。

当前文件数量还不多，所以现在不强制迁移。

### 建议 4：为重大架构变更预留 ADR 类目录

如果后续开始频繁出现“为什么做这个架构选择”的记录，建议新增：

- `docs/adr/`

用于存放：

- 关键设计决策
- 取舍背景
- 方案对比

它和 `core` 的区别是：

- `core` 写“系统现在是什么”
- `adr` 写“为什么最终变成这样”

## 11. 推荐的长期目标结构

如果文档继续增长，建议最终演进到：

```text
docs/
├── README.md
├── core/
│   ├── architecture.md
│   ├── api.md
│   ├── database.md
│   └── security.md
├── interface/
│   └── design.md
├── features/
│   ├── features-guide.md
│   ├── gallery.md
│   ├── map.md
│   ├── timeline.md
│   ├── selection.md
│   ├── content-experience.md
│   ├── usage-monitoring.md
│   └── admin/
├── guide/
│   ├── documentation.md
│   ├── standards.md
│   ├── tech-stack.md
```

注意：

- 这是一条演进目标，不是要求现在全部创建
- 当前体量下，保留四个主目录已经足够

## 12. 文档维护清单

每次做完重要改动后，至少检查以下问题：

1. 这次改动影响的是 `core`、`features`、`interface` 还是 `guide`？
2. 是否已经有现成文档可以承接，而不是再新建一篇？
3. 文档里的“最后更新”是否已改成真实日期？
4. 文档里的规则是否和代码当前行为一致？
5. 如果一个能力横跨多个文档，是否至少同步了入口索引？

## 13. 一句话结论

当前文档分类基础是合适的，不需要推倒重来。

真正需要持续执行的是：

- `README` 只做索引
- `core / interface / features / guide` 严守边界
- 大文档及时拆分
- 规则改动同步更新对应文档
